# Don’t Regress Into Clutter — Checklist

Use this checklist before finishing any work package.

## Hierarchy

- Is there only one dominant hero or proof object?
- Are supporting figures clearly subordinate?
- Does the strongest evidence still read as the strongest evidence?

## Framing and chrome

- Is any panel/chrome louder than the figure or argument it contains?
- Did callouts, pills, badges, or dividers multiply unnecessarily?
- Did repeated cards create a dashboard feel?

## Substance

- Did every premium treatment earn its place through real evidence?
- If a new visual is striking, does it reveal something technically meaningful?
- Were diagnostics kept visible where they are part of the proof?

## Narrative integrity

- Is the page clearer about problem, method, evidence, and tradeoffs?
- Did the wording become more promotional than technical?
- Were any technical-content changes beyond readability explicitly reported?

## System stability

- Did dark mode remain coherent?
- Did mobile hierarchy remain coherent?
- Did a shared style change make API or Quickstart louder?
- Are any new classes/components durable and easy to stabilize?

## Page-specific checks

### Homepage
- one dominant signature visual only
- routes remain obvious
- no dashboard wall

### Surface repair
- slice-level evidence remains visible
- pretty surface did not replace reviewability

### eSSVI
- assumptions and tradeoffs are easier to see
- diagnostics remain present

### Architecture
- the diagram carries more authority on its own
- surrounding framing did not become the solution

### Local-vol / PDE
- repricing/validation logic remains primary
- 3D, if used, is supporting only

### Performance
- benchmark interpretation improved
- no pseudo-product dashboard creep
